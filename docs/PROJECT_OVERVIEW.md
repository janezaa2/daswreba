# პროექტის მიმოხილვა

## დანიშნულება

`daswreba` არის მრავალკომპანიური დასწრების აღრიცხვის სისტემა. ის ფიზიკურ fingerprint მოწყობილობას ცვლის ვებ და Android check-in პროცესით: მოლარე შედის უნიკალური კოდით, ირჩევს სალაროს, აზიარებს GPS მდებარეობას და აფიქსირებს მიმდინარე დღის დასწრებას.

თითოეული კომპანია საკუთარ სივრცეში მართავს მოლარეებს, სალაროებს, დაშვებულ ლოკაციებს, დასწრების ჟურნალსა და პარამეტრებს. პლატფორმის ადმინისტრატორი მართავს კომპანიებს, ადმინისტრატორებსა და საჯარო საიტის კონტენტს.

## ძირითადი შესაძლებლობები

- კომპანიების თვითრეგისტრაცია და პლატფორმის ადმინისტრატორის მიერ მათი გააქტიურება;
- კომპანიების მონაცემების ერთმანეთისგან გამიჯვნა (`companyId`-ით);
- მოლარეებისა და სალაროების შექმნა, შეცვლა და გააქტიურება/დეაქტივაცია;
- ვებ და Android აპიდან დასწრების დაფიქსირება;
- კომპანიის რამდენიმე დაშვებული GPS ლოკაციისა და რადიუსის მართვა;
- ლოკაციის ფარგლებს გარეთ შესრულებული check-in-ის მონიშვნა;
- ერთი მოლარისთვის დღეში მხოლოდ ერთი ჩანაწერის დაშვება;
- ერთი მოწყობილობით იმავე დღეს სხვადასხვა მოლარის check-in-ის შეზღუდვა;
- დასწრების ფილტრაცია და Excel ფაილად გატანა;
- `About` და `Contact` გვერდების კონტენტის მართვა.

## მომხმარებლები და წვდომა

### პლატფორმის ადმინისტრატორი

როლი: `platform_admin`

- შედის `/login` გვერდიდან;
- მუშაობს `/admin/*` სივრცეში;
- მართავს კომპანიებს, ადმინისტრატორებსა და საიტის კონტენტს;
- შეუძლია pending კომპანიის გააქტიურება ან კომპანიის დეაქტივაცია.

### კომპანიის მომხმარებელი

როლი: `company_user`

- შედის `/login` გვერდიდან;
- მუშაობს `/company/*` სივრცეში;
- ხედავს და ცვლის მხოლოდ საკუთარ კომპანიასთან დაკავშირებულ მონაცემებს;
- მართავს მოლარეებს, სალაროებს, ლოკაციებს, დასწრებასა და კომპანიის პარამეტრებს.

### მოლარე

- ვებ check-in-ს იწყებს `/check-in` გვერდიდან უნიკალური კოდით;
- Android აპში კოდის დადასტურების შემდეგ იღებს Bearer token-ს;
- ირჩევს მხოლოდ საკუთარ კომპანიაში არსებულ აქტიურ სალაროს;
- დასწრების დასაფიქსირებლად აგზავნის მდებარეობასა და მოწყობილობის იდენტიფიკატორს.

ადმინისტრატორის სესია ინახება HTTP-only `admin_session` cookie-ში. `/admin/*` და `/company/*` მარშრუტებს როლის მიხედვით `src/proxy.ts` იცავს. API endpoint-ები დამატებით ამოწმებს როლს, კომპანიის სტატუსსა და `companyId`-ს.

## check-in-ის პროცესი

1. მოლარის კოდი მოწმდება და შესაბამისი აქტიური მოლარე იძებნება.
2. სისტემა აბრუნებს ამავე კომპანიის აქტიურ სალაროებს.
3. კლიენტი აგზავნის არჩეულ სალაროს, GPS კოორდინატებსა და `deviceId`-ს.
4. სერვერი ამოწმებს, რომ სალარო მოლარის კომპანიას ეკუთვნის.
5. სერვერი კრძალავს იმავე მოლარის მეორე check-in-ს მიმდინარე დღეს.
6. სერვერი კრძალავს იმავე მოწყობილობის გამოყენებას სხვა მოლარის მიერ მიმდინარე დღეს.
7. ჩართული geofence-ის შემთხვევაში კოორდინატები მოწმდება კომპანიის ყველა დაშვებულ ლოკაციასთან.
8. დასწრება ინახება `AttendanceRecord`-ში, წყაროს მითითებით (`web` ან მობილური პლატფორმა).

geofence-ის ფარგლებს გარეთ ყოფნა ჩანაწერს არ ბლოკავს: ჩანაწერს ენიჭება `outOfRange = true`, ხოლო ვებ კლიენტი იღებს გაფრთხილებას უახლოეს დაშვებულ ტერიტორიამდე მიახლოებითი მანძილით.

## არქიტექტურა

პროექტი არის ერთიანი Next.js full-stack აპლიკაცია:

- **UI და გვერდები:** Next.js App Router და React კომპონენტები `src/app`-სა და `src/components`-ში;
- **HTTP API:** Route Handlers `src/app/api`-ში;
- **ავტორიზაცია:** JWT ხელმოწერა `jose`-ით, ადმინისტრატორის cookie სესია და მოლარის mobile Bearer token;
- **მონაცემები:** PostgreSQL და Prisma ORM;
- **ვალიდაცია:** Zod სქემები API მოთხოვნებისთვის;
- **გეოლოკაცია:** Leaflet რუკისთვის და Haversine ფორმულა სერვერული მანძილის შემოწმებისთვის;
- **ექსპორტი:** `xlsx` დასწრების Excel ფაილად გასატანად;
- **სტილები:** Tailwind CSS 4.

### საქაღალდეების დანიშნულება

```text
prisma/
  schema.prisma      მონაცემთა მოდელები და კავშირები
  migrations/        PostgreSQL მიგრაციები
  seed.ts            ლოკალური სატესტო მონაცემები
src/
  app/                გვერდები, layout-ები და API route-ები
  components/         UI, ადმინისტრირების და check-in კომპონენტები
  lib/                auth, Prisma, თარიღი, geofence და API დამხმარეები
  types/              საერთო TypeScript ტიპები
  proxy.ts            როლებზე დაფუძნებული გვერდების დაცვა
public/                სტატიკური ფაილები
```

## მონაცემთა მოდელი

- `Company` — tenant-ის ძირითადი ჩანაწერი და მისი სტატუსი;
- `CompanyLocation` — კომპანიის დაშვებული GPS ცენტრი და რადიუსი;
- `AdminUser` — პლატფორმის ან კომპანიის ადმინისტრაციული მომხმარებელი;
- `Cashier` — კომპანიის მოლარე და მისი უნიკალური კოდი;
- `CashRegister` — კომპანიის სალარო;
- `AttendanceRecord` — მოლარის დღიური check-in, მდებარეობა, მოწყობილობა და პლატფორმა;
- `SiteContentBlock` — საჯარო `About` და `Contact` გვერდების კონტენტი.

`AttendanceRecord` ინახავს მოლარისა და სალაროს სახელების ასლებსაც. ამიტომ ისტორიული ჩანაწერი გასაგები რჩება მაშინაც, თუ შესაბამისი მოლარის ან სალაროს სახელი მოგვიანებით შეიცვლება.

## ლოკალური გაშვება

საჭიროა Node.js/npm და ხელმისაწვდომი PostgreSQL ბაზა.

1. დააყენეთ დამოკიდებულებები:

   ```bash
   npm install
   ```

2. პროექტის root-ში შექმენით `.env`:

   ```dotenv
   DATABASE_URL="postgresql://USER:PASSWORD@HOST:5432/DATABASE?schema=public"
   JWT_SECRET="replace-with-a-long-random-secret"
   ```

3. გაუშვით მიგრაციები და სურვილის შემთხვევაში seed:

   ```bash
   npm run db:migrate
   npm run db:seed
   ```

4. გაუშვით development server:

   ```bash
   npm run dev
   ```

5. გახსენით `http://localhost:3000`.

### გარემოს ცვლადები

| ცვლადი | აუცილებელი | დანიშნულება |
|---|---:|---|
| `DATABASE_URL` | დიახ | PostgreSQL კავშირის სტრიქონი |
| `JWT_SECRET` | დიახ | ადმინისტრატორისა და მოლარის JWT-ების ხელმოწერა |
| `NEXT_PUBLIC_ANDROID_APK_URL` | არა | Android APK-ის ჩამოტვირთვის ალტერნატიული URL |
| `SEED_PLATFORM_ADMIN_USERNAME` | არა | seed პლატფორმის ადმინისტრატორის username |
| `SEED_PLATFORM_ADMIN_PASSWORD` | არა | seed პლატფორმის ადმინისტრატორის password |
| `SEED_COMPANY_ADMIN_USERNAME` | არა | seed კომპანიის მომხმარებლის username |
| `SEED_COMPANY_ADMIN_PASSWORD` | არა | seed კომპანიის მომხმარებლის password |

## seed მონაცემები

თუ seed გარემოს ცვლადები არ არის მითითებული, `npm run db:seed` ქმნის:

- პლატფორმის ადმინისტრატორს: `superadmin` / `super123`;
- კომპანიის მომხმარებელს: `admin` / `admin123`;
- სატესტო კომპანიას, სამ სალაროს და სამ მოლარეს;
- მოლარის კოდებს: `100001`, `100002`, `100003`.

ეს მონაცემები განკუთვნილია მხოლოდ ლოკალური development გარემოსთვის. production-ში გამოიყენეთ უნიკალური ძლიერი პაროლები და seed-ის default ანგარიშები არ დატოვოთ.

## ცვლილებებზე მუშაობა

- ახალი სამუშაო დაიწყეთ `main`-იდან შექმნილ ცალკე branch-ზე;
- ცვლილებამდე ჩამოიტანეთ ბოლო `origin/main`;
- commit-მდე გაუშვით მინიმუმ `npm run lint` და საჭიროების შემთხვევაში `npm run build`;
- remote branch ატვირთეთ upstream კავშირის შექმნით: `git push -u origin <branch>`;
- `main`-ში ცვლილება შეიტანეთ review/merge პროცესის შემდეგ.
